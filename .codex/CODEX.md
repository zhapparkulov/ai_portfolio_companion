# CODEX.md — правила для Codex

Codex работает в репозитории `ai_portfolio_companion` как инженерный ассистент.
Главная задача — делать изменения маленькими, проверяемыми и совместимыми с
архитектурой проекта.

## Источники правды

1. [`../docs/ARCHITECTURE.md`](../docs/ARCHITECTURE.md) — архитектура приложения.
2. [`../.claude/CLAUDE.md`](../.claude/CLAUDE.md) — общие правила, слои, тесты и стиль.
3. [`./changes.md`](./changes.md) — история решений и изменений Codex.

Если правила конфликтуют, приоритет такой:

1. Явный запрос пользователя.
2. `docs/ARCHITECTURE.md`.
3. `.claude/CLAUDE.md`.
4. `.codex/CODEX.md`.

## Как работать

- Перед структурными изменениями читать архитектуру и соседний код.
- Не создавать верхнеуровневую `lib/`; Flutter-код живет только в `mobile_app/lib/`.
- Соблюдать направление зависимостей: `presentation -> domain <- data`.
- Не смешивать несколько feature-slice в один большой бесконтрольный diff.
- После нетривиальных изменений запускать релевантные проверки:
  - `dart format`
  - `flutter analyze`
  - `flutter test`
- В конце сессии обновлять [`changes.md`](./changes.md).

## Что записывать в `changes.md`

Записывай только полезное:

- дата и краткое название сессии;
- что изменилось;
- какие файлы или слои затронуты;
- какие решения приняты;
- какие проверки запускались;
- что осталось следующим шагом.

Не нужно копировать весь diff. Журнал должен помогать быстро понять историю,
а не заменять git.

## Формат короткого architecture impact

В конце крупных изменений добавляй блок:

```text
Architecture impact:
- Слои:
- Почему так:
- Риск:
- Проверки:
```
